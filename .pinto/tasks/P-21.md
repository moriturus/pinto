# **Summary**

Allow label-setting CLI options to accept multiple label values after one flag, including `add --label foo bar` and `edit <id> --label foo bar`, and apply the same behavior to any other option that mutates labels.

# **Implementation Plan**

1. Configure the mutating `add` and `edit` label arguments to consume one or more values per occurrence while retaining Clap's existing append behavior for repeated `--label` flags. Leave the `list` and `board` label-filter arguments unchanged.
2. Add CLI integration coverage that first demonstrates multi-value parsing for `add`, replacement semantics for `edit`, generated help, and compatibility with repeated label flags and label filters. Keep the Rust argument help and both English and Japanese Fluent help synchronized, using a singular `LABEL` value name plus an ellipsis to communicate the multi-value shape.
3. Update the root README, mdBook CLI/cookbook guidance, and the persisted `labels` demo README with the supported compact multi-label syntax.
4. Verify targeted tests, dogfood the resulting commands with `cargo run`, complete the iterative review loop, and run the repository's full `mise run check` gate.

# **Acceptance Criteria**

- [x] `pinto add "Example" --label foo bar` stores both `foo` and `bar` on the new PBI.
- [x] `pinto edit <id> --label foo bar` replaces the PBI's labels with both `foo` and `bar`.
- [x] Every other existing CLI option that sets or replaces labels, if any, accepts the same variable-number syntax and receives aligned help and test coverage.
- [x] Existing repeated `--label` usage and label-filtering options remain compatible and are not changed into label-setting operations.
- [x] CLI tests and user-facing examples document the supported multi-label syntax.
