# **Summary**

Independent clones of a shared board issue sequential IDs locally, so parallel work can allocate the same ID; after a Git merge the board holds distinct items sharing one ID. `doctor` reports `DuplicateId` but cannot repair it, leaving manual file surgery as the only recovery.

# **Implementation Plan**

1. Extend the file/Git doctor repair path to group valid duplicate PBI records in deterministic order: active tasks first, archived items second, then lexical path order. The first occurrence keeps its ID; later occurrences receive fresh IDs above every existing or historically issued number for the same prefix.
2. Treat each duplicate occurrence index as the deterministic merge lineage. When a repaired duplicate record refers by `parent` or `depends_on` to another duplicated ID with the same occurrence index, rewrite that reference to the corresponding fresh ID; references without an unambiguous matching occurrence remain attached to the canonical first record.
3. Rewrite each repaired record through the normal Markdown serializer, move it to the canonical `<new-id>.md` path in its original active/archive area, and append every new ID to `issued_ids`. When duplicate repair changes the board, re-inspect before applying the existing filename/history fixes so a surviving non-canonical filename is normalized in the same run; the final doctor scan verifies the result. SQLite needs no duplicate repair because its primary key prevents duplicate IDs.
4. Add Red-first service and CLI regression tests for duplicates within tasks, within archive, and across both areas after a simulated merge. Update the merge runbook/demo guidance to make `doctor --fix` the documented recovery path, then run the full project checks and dogfood the command on a disposable copied board.
