# **Summary**

`export --json` has no inverse: restoring a board from a snapshot or migrating from another tool requires hand-written automation plans. Add an import that recreates board data from the exported JSON contract.

# **Acceptance Criteria**

- [x] A documented command restores PBIs, Sprints, configuration, and the shared DoD from an `export --json` snapshot.
- [x] Importing into a non-empty board fails fast unless an explicit mode is chosen.
- [x] A round-trip test proves that export followed by import reproduces an equivalent board.
- [x] A feature decision record covers persistence and compatibility impact.


# **Implementation Plan**

## Command surface

Add a top-level `import` subcommand, the inverse of `export --json`.

- `pinto import <SOURCE>` where `SOURCE` is a path to an `export --json`
  document, or `-` to read the JSON from standard input (mirrors
  `automate`'s source handling).
- `--force` / `-f` opts into replacing an existing non-empty board. Without
  it, importing into a board that already has PBIs or Sprints fails fast.

## Data flow (inverse of export)

`export --json` builds a `BoardSnapshot` (domain types) and the CLI serializes
it through DTOs in `cli/json.rs`. Import is the exact mirror:

1. CLI reads the JSON text and parses it back into a `BoardSnapshot` via a new
   `parse_export` in `cli/json.rs`. The export DTOs gain `Deserialize` plus
   converters that turn the string fields back into `ItemId`, `Rank`, `Status`,
   `SprintId`, `SprintState`, and RFC3339 `DateTime<Utc>`. `config` stays a
   `serde_json::Value`; `dod` stays `Option<String>`.
2. The service `import_board(project_dir, snapshot, force)` writes the snapshot
   to the board, deserializing `config` into a `Config` and validating it.

## Service behavior

`import_board` requires an initialized board (`Error::NotInitialized`
otherwise) and serializes the whole operation under the board write lock, like
`migrate`.

- The target backend is the one the current board config selects. Emptiness is
  checked against that backend: if it holds any active PBIs or Sprints and
  `force` is false, return `ImportOutcome::Refused { items, sprints }` (the CLI
  maps this to exit code 1 with actionable guidance). This keeps the domain
  `Error` set stable — refusal is normal control flow, following the
  `MigrateOutcome` precedent.
- With `force` (or an empty board): mirror the snapshot. Delete every existing
  active PBI and Sprint, then save the snapshot's items and Sprints (saving an
  item records its issued ID, so future `add` never reuses IDs). Overwrite
  `config.toml` with the snapshot config and write (or clear) `dod.md`.
- Commit once through the backend so the Git backend records the restore.
- Returns `ImportOutcome::Imported { items, sprints }`.

## Round-trip equivalence

`export`'s JSON contract omits Sprint capacity fields (daily hours, holidays,
deduction factor), so byte-identical domain equality is not the right measure.
Equivalence is defined against the contract: `export → import → export`
produces identical JSON. The integration test asserts that.

## Compatibility / persistence

No backward-compatibility constraints. Import reuses the existing plain-text
persistence and the stable `export --json` schema; the contract is documented in
`docs/json-schema.md`. A feature decision record is added to
`docs/book/src` (and any maintained-guidance doc) covering the persistence and
compatibility impact, satisfying AC #4.
