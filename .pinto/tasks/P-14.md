# **Summary**

Add Sprint and label filters to the interactive Kanban view so the TUI can focus on the active work scope.

# **Implementation Approach**

## CLI contract

- Add `--sprint`, `--label`, and `--all-labels` to `kanban`, reusing the board command's exact Sprint and label matching semantics.
- Keep `--column` as the display-column scope and keep all startup filters read-only.

## Filter propagation

- Build one `BoardQuery` at the CLI boundary with Sprint, labels, label matching, and startup search, then pass it into the Kanban runtime.
- Store the query scope with the view so every reload, edit, move, reorder, and live-search operation reapplies the startup filters. Live search changes only the query's search component, preserving Sprint and labels.
- Continue filtering rendered columns after loading the query-scoped board so column selection composes with all item filters and preserves cross-column metadata for the filtered scope.

## Verification

- Follow Red → Green → Refactor with argument/help tests, runtime board-loading tests for Sprint and label matching, composition tests for search and regex, and a PTY startup-filter test.
- Update the Kanban documentation and examples to describe the new startup filters and their non-mutating behavior.

# **Acceptance Criteria**

- [x] `pinto kanban --sprint <id>` displays only PBIs assigned to the requested Sprint.
- [x] `pinto kanban --label <label>` applies the same label-matching behavior used by the board view.
- [x] Sprint and label filters compose with `--column`, `--search`, and `--regex`.
- [x] Startup filters do not modify board data and are described in command help and documentation.
