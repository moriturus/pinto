Summary

Separate personal keybindings from shared board configuration and document configuration compatibility.

Implementation Plan

### Configuration ownership
- Remove keybindings from the shared `.pinto/config.toml` schema and from `pinto init` output.
- Keep the existing `[tui.key_bindings]` TOML shape in the per-user file at `$XDG_CONFIG_HOME/pinto/config.toml`; use the standard home configuration fallback when `XDG_CONFIG_HOME` is unset.

### Loading and validation
- Load the optional user file when Kanban settings are requested, defaulting to built-in bindings when it is absent.
- Apply strict unknown-field, action, assignment, and key-expression validation and report the user-config path in parse errors.

### Compatibility guidance
- Update the configuration, Kanban, data-format, README, demo, changelog, and stability guidance so board TOML, Markdown, SQLite, and JSON contracts are clearly separated.
- Explain that newer board configuration keys can reject older binaries and that downgrade recovery requires restoring/removing those keys, while personal keybindings stay outside the board.

### Verification
- Add unit and CLI integration coverage for default, custom, invalid, and board-isolation behavior.
- Run the repository quality gates, dogfood the new configuration through `cargo run`, complete the acceptance criteria, and commit with the P-26 identifier.

Acceptance Criteria

- [x] Keybindings are stored at `$XDG_CONFIG_HOME/pinto/config.toml`.
- [x] Keybindings are not stored in `.pinto/config.toml`, so personal preferences do not become shared board state.
- [x] Compatibility guidance distinguishes board configuration from Markdown, SQLite, and JSON data formats.
- [x] Releases that add board configuration keys state whether older binaries can read the updated configuration and provide downgrade guidance when necessary.
