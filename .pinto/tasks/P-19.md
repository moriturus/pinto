# **Summary**

Allow pinto commands to discover the nearest ancestor `.pinto` board when invoked from a repository subdirectory, with an explicit override for scripts.

# **Implementation Plan**

- Resolve the project directory at CLI startup with precedence `--dir`, `PINTO_DIR`, then ancestor discovery from the current directory.
- Accept an override that names either a project directory containing `.pinto` or the `.pinto` directory itself, and normalize it before dispatch.
- Search the current directory first, then parents; check the filesystem root and stop after a repository root marked by `.git`, documenting that boundary.
- Preserve `pinto init` behavior by initializing the explicitly selected/current directory without ancestor auto-discovery.
- Keep caller-owned relative automation plan paths anchored to the invocation directory and restore the shell working directory after each dispatch so board discovery does not leak context.
- Add CLI integration coverage for nearest-ancestor selection, explicit flag/environment overrides, root behavior, actionable no-board errors, and relative automation plans; add a nested discovery demo.

# **Acceptance Criteria**

- [x] Commands issued from a descendant directory locate and use the nearest ancestor `.pinto` board.
- [x] An explicit board-directory override is available through `--dir` or `PINTO_DIR` for scripts and agents.
- [x] Searching stops at the filesystem root or documented repository boundary and reports an actionable error when no board is found.
- [x] Invoking pinto from the board root keeps the current behavior.
