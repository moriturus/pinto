# **Summary**

Keep pinto lightweight while preserving SQLite as a supported, opt-in storage backend.

## Implementation plan

### 1. Preserve the feature boundary

Make the default Cargo feature set explicit and keep `rusqlite` optional behind the `sqlite` feature. Document how the default file backend, the Git backend, and the opt-in SQLite backend differ.

### 2. Record persistence and compatibility guarantees

State that file and Git backends retain human-readable Markdown/TOML data and Git-diff workflows, while SQLite is the deliberate normalized-storage exception. Keep the existing versioned-schema recovery guidance and make future schema changes require an explicit migration plan.

### 3. Make feature decisions reviewable

Add a design decision checklist for new commands, backends, and reports covering Scrum need, insufficiency of existing functionality, dependency impact, persistence impact, and migration/compatibility impact. Verify the feature boundary and the file-to-SQLite-to-file behavior with focused CLI integration tests and a runnable demo; keep prose documentation free to evolve without turning its exact wording into an API.

# **Acceptance Criteria**

- [x] SQLite support remains available through the non-default `sqlite` feature.
- [x] The default build does not enable the SQLite dependency, and the feature boundary remains documented.
- [x] Plain-text and Git-diff guarantees are stated for file and Git backends, with SQLite described as an explicit exception.
- [x] New commands, backends, and reports require a Scrum-related need, an explanation of why existing functionality is insufficient, and a record of dependency, persistence, and migration impact.
- [x] Future SQLite schema work improves migration and compatibility guidance without making SQLite support or its non-default status a removal target.
