# **Summary**

Make backend behavior and Markdown serialization contracts explicit across supported feature configurations.

## Implementation Approach

1. Add one shared integration-test harness over the public backend facade. Run the same item and sprint persistence workflows against file, Git, and SQLite when the feature is enabled.
2. Compare loaded/listed domain values and stable `Error::code()` categories across backends. Keep Git commit-boundary assertions and SQLite schema assertions in backend-specific tests.
3. Extend the Markdown module with `proptest` round-trip properties for valid PBI and Sprint values, including Unicode text, optional fields, UTC timestamps, and relationship IDs.
4. Keep the SQLite-disabled path explicit with a `cfg(not(feature = "sqlite"))` configuration test and verify the default toolchain path separately.
5. Add a small `demos/single/storage-contract` board and README that exercise the same persistence contract from the CLI. No production behavior change is expected unless a contract test exposes one.

# **Acceptance Criteria**

- [x] Shared contract tests cover item and sprint workflows across file, Git, and SQLite backends.
- [x] Contract tests compare domain results and error categories while backend-specific commit and schema checks remain separate.
- [x] The disabled-SQLite feature path is tested with the current toolchain.
- [x] Property tests verify Markdown round trips for backlog items and sprints, including Unicode, optional values, timestamps, and relationship IDs.
