# **Summary**

`pinto doctor` inspects the complete board twice on every run: once before and once after the fix stage, even without `--fix` or when no safe fix was applied. The second inspection doubles I/O and parsing for the common healthy-board case.

# **Acceptance Criteria**

- [x] `doctor` without `--fix` performs exactly one full inspection.
- [x] `doctor --fix` re-inspects only when at least one fix was applied.
- [x] A regression test covers the single-inspection path and the existing doctor tests pass.

# **Implementation Plan**

## Cause

`run_doctor` in `src/service/doctor.rs` unconditionally runs a final `inspect_board` after the fix stage, so plain `doctor` and `--fix` runs that repair nothing both pay two full board inspections.

## Approach

1. Extract the inspect/fix/report control flow into `run_doctor_with`, which takes the inspection step as an injected closure. Production code passes `inspect_board`; unit tests pass a counting wrapper to assert the exact inspection count.
2. Without `--fix`, report the issues from the single initial inspection and return.
3. With `--fix`, keep the existing post-renumber re-inspection (filename and issued-history repairs must see the renumbered board), and reuse the latest inspection for the final report when the safe-fix stage applied nothing. Re-inspect at the end only when at least one safe fix changed the board after the last inspection.
4. Resulting inspection counts: plain `doctor` 1; `--fix` with nothing applied 1; duplicate renumber only 2; safe fixes only 2; renumber followed by safe fixes 3.

## Tests

- Unit (service, TDD Red first): count inspections through `run_doctor_with` for the scenarios above.
- Integration (CLI): `doctor --fix` on a board whose only issues are not auto-fixable reports the issues without any `Fixed:` line, pinning the fix-mode single-inspection path.

## Demos

Extend `demos/single/doctor/README.md` to state that `doctor` runs a single inspection unless a repair was applied.

## Outcome

Implemented as planned with no deviations. Verified through the counting unit tests (1/1/2/2 inspections), the new CLI regression test, `mise run check`, and dogfooding on the project board plus a scratch copy of the doctor demo (repair path re-inspects and reports the repaired board).
