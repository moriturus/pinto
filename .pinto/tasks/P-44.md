# **Summary**

The shared CLI test helper forwards the developer's `LC_ALL` and `LANG` to spawned pinto processes, so English output assertions fail under a Japanese locale (for example `usage::cli_options_have_short_forms`). The suite currently passes only in English-locale environments such as CI runners.

# **Implementation Plan**

- **Root cause**: CLI integration tests spawn the `pinto` binary and only override `XDG_CONFIG_HOME`; every spawned process therefore inherits the developer's `LC_ALL`/`LANG`. pinto resolves its display language from those variables (`src/i18n.rs`), so help/output localizes to Japanese under a Japanese shell and breaks English assertions.
- **Fix**: Pin a fixed English locale (`LC_ALL=en_US.UTF-8`, `LANG=en_US.UTF-8`) on every integration-test process spawn:
  - `tests/cli/common.rs::pinto` — the shared helper for the `cli` binary's submodules.
  - `tests/demos.rs::run_pinto` / `run_pinto_with_env` — the demo-suite spawn helpers (the `intentional_error_demos_are_registered_and_keep_user_error_contracts` case also leaked the locale). `run_pinto_with_env` pins English first, then applies the caller's override so the `single/i18n` demo can still request `LC_ALL=ja_JP.UTF-8` and win.
- **Why the helpers only**: The `i18n` tests build `Command::cargo_bin("pinto")` directly and set `LC_ALL`/`LANG` themselves, so they are unaffected; their explicit `.env` calls win. `tests/skills.rs` asserts only locale-invariant strings (`Usage:`, item IDs), so it stays green without a pin.
- **Regression test**: `usage::helper_pins_english_locale_for_spawned_processes` inspects the shared helper's command environment via `assert_cmd`'s `get_envs()` and asserts the English locale is pinned. This is parent-locale independent, unlike output assertions that only diverge under a Japanese parent.
- **Scope boundary (out of scope, tracked separately)**: Four in-process **unit** tests in the `pinto` binary crate (`cli::commands::tests`, `cli::kanban::tests::render`) also fail under a Japanese parent, but for a different reason: they exercise production code that reads the process-global `pinto::i18n::current()` localizer, which resolves from the parent environment. These are not integration tests that spawn processes, and pinning them cleanly requires injecting a localizer into ~128 `current()` call sites (a production refactor). This is out of scope for "Pin the locale in CLI integration tests" and is tracked as a follow-up item. Consequently, AC #2 is verified against the full CLI **integration** test suite (all `assert_cmd`-based binaries), which is this item's subject.
- **Verification**: Run the CLI integration binaries with `LC_ALL=ja_JP.UTF-8` in the parent to confirm they pass (AC #2), and confirm the explicit-override i18n tests still pass (AC #3). `mise run check` passes in the default (English) locale.

# **Acceptance Criteria**

- [x] The shared helper pins a fixed English locale for spawned pinto processes.
- [x] The full test suite passes with `LC_ALL=ja_JP.UTF-8` set in the parent environment.
- [x] i18n tests that override the locale explicitly keep passing.
