# **Summary**

On the self-hosted board, `pinto doctor` reports rank anomalies where two items share the same rank within one status/parent scope (observed for the done column: P-24/P-43 and P-28/P-42) after routine `move` and completion operations. `doctor` recommends `pinto rebalance`, but `rebalance` only compresses oversized rank *length* and reports `Ranks already balanced` when duplicates exist, so the suggested repair cannot resolve the finding; `pinto reorder` is currently the only manual workaround. Investigate whether rank assignment during `move`/`reorder`/completion can hand out a rank already used in the destination scope, fix the root cause, and reconcile doctor's repair guidance with a command that actually resolves duplicates.

# **Acceptance Criteria**

- [x] The root cause of duplicate ranks within a single status/parent scope is identified and documented, including which operation(s) (`move`, `reorder`, completion) can assign a rank already present in the destination scope.
- [x] Rank assignment no longer produces duplicate ranks within a single status/parent scope during normal operations, covered by a regression test.
- [x] `pinto doctor`'s rank-anomaly repair guidance names a command that actually resolves duplicate ranks (extend `pinto rebalance` to handle duplicates or point to the correct command), and behavior matches the guidance.
- [x] `pinto doctor` reports the board healthy after the covered scenarios.

# **Implementation Plan**

**Root cause (AC1):** rank values are unique only within a `(status, parent)` sibling scope, never across the board. `add` and `reorder` always mint an in-scope rank, but `move` (`transition_to` in `src/backlog/item.rs`) carries the rank across a status change untouched. When the destination scope already holds that value, the move creates an in-scope duplicate — the observed P-24/P-43 and P-28/P-42 pairs in `done`. Cross-scope equal values are common because a per-scope `rebalance` (`Rank::rebalance(sibling_count)`, a pure function of the count) hands equal-size scopes byte-identical sequences. Separately, `rebalance()`'s rewrite decision only checked rank *length* (`current.max_len <= target.max_len`), so a scope already holding a same-length literal duplicate (e.g. two `"i"`) was skipped and reported "already balanced" even though `doctor` flagged it — the suggested repair could not resolve the finding.

**Fix (AC2/AC3):**
1. Enforce the invariant where a collision is born: `move_item_with_outcome` (`src/service/item/crud.rs`) keeps a transitioned item's rank so its position travels with it, but if that rank already exists in the destination `(status, parent)` scope it re-pegs the item after the backlog's maximum rank (`Rank::after(backlog_max)`, like `add`) so it lands at the destination tail with a unique value. A carried duplicate is the only way a normal operation could break the invariant, so this closes it without disturbing non-colliding moves. (AC2)
2. Keep `rebalance()` per-scope and shortest-width, but broaden its rewrite trigger to fire on length-bloat *or* the presence of a literal duplicate rank, so `doctor`'s recommended `pinto rebalance` actually repairs a scope corrupted by manual edits. The rewrite width is checked with `Rank::rebalance_width` without allocating the sequence. (AC3)
3. `doctor.rs`'s repair guidance ("run pinto rebalance for the affected workflow scope") is left unchanged — it names the correct command and is accurate once `rebalance()` repairs literal duplicates.

**Tests (TDD):** unit tests in `src/service/item/tests/reorder.rs` cover the move re-peg (a moved item does not reuse a rank already held in the destination scope, lands at its tail, and leaves `doctor` healthy) and that a same-status move preserves the rank; `rebalance_rewrites_scope_with_literal_duplicate_rank_at_canonical_length` and `doctor_reports_healthy_after_rebalance_resolves_duplicate_ranks` cover the duplicate-repair path; `src/rank.rs` covers `rebalance_width`. A CLI integration test (`tests/cli/move_reorder.rs::move_does_not_duplicate_a_rank_already_held_in_the_destination_scope`) reproduces the P-24/P-43 collision end-to-end through the `pinto` binary and was confirmed to fail against the pre-fix code.
