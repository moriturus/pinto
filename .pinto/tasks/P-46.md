# **Summary**

On the self-hosted board, `pinto doctor` reports rank anomalies where two items share the same rank within one status/parent scope (observed for the done column: P-24/P-43 and P-28/P-42) after routine `move` and completion operations. `doctor` recommends `pinto rebalance`, but `rebalance` only compresses oversized rank *length* and reports `Ranks already balanced` when duplicates exist, so the suggested repair cannot resolve the finding; `pinto reorder` is currently the only manual workaround. Investigate whether rank assignment during `move`/`reorder`/completion can hand out a rank already used in the destination scope, fix the root cause, and reconcile doctor's repair guidance with a command that actually resolves duplicates.

# **Acceptance Criteria**

- [ ] The root cause of duplicate ranks within a single status/parent scope is identified and documented, including which operation(s) (`move`, `reorder`, completion) can assign a rank already present in the destination scope.
- [ ] Rank assignment no longer produces duplicate ranks within a single status/parent scope during normal operations, covered by a regression test.
- [ ] `pinto doctor`'s rank-anomaly repair guidance names a command that actually resolves duplicate ranks (extend `pinto rebalance` to handle duplicates or point to the correct command), and behavior matches the guidance.
- [ ] `pinto doctor` reports the board healthy after the covered scenarios.
