# **Summary**

Every file-backend operation reads and parses the complete task and archive set before acting, so command latency grows linearly with board size, including single-item reads. The fail-fast validation contract in `docs/stability.md` motivates this, but the latency curve on large boards has never been measured.

## **Implementation plan**

### Benchmark harness

Add a dependency-light `large_board_bench` Rust example. It will generate an export-compatible snapshot outside `.pinto`, initialize a temporary board, restore the generated 1,000-item and 10,000-item boards through `cargo run`, and measure `list`, `show`, `add`, and `move` through the same `cargo run` path. A warm-up invocation will remove first-build noise; each reported value will be the median of repeated samples, with command output captured so rendering does not dominate the measurement.

### Scaling decision

Keep the file backend's complete task and archive validation contract. The benchmark will document the observed linear scaling, while single-item reads will not bypass validation in this item: allowing malformed unrelated records to remain invisible would weaken the fail-fast integrity guarantee. A future optimization would require a separate compatibility and repair decision.

### Verification and documentation

Add a CLI integration test covering the generated-board workflow, a small checked-in demo board with a README, and record the measured environment, method, and results in `docs/benchmarks.md`. Update `docs/stability.md` with the explicit read-scaling decision, then run the focused tests and `mise run check` before dogfooding the completed item.

### Implementation update

The first smoke run showed that an example launched by `cargo run` cannot launch nested `cargo run` commands: the outer Cargo process holds the build-directory lock while the example is running. The example will therefore be built once by `scripts/large-board-benchmark.sh` and launched directly; it will then invoke each pinto operation as a separate `cargo run` child after the build lock is released. This preserves the required dogfooding path without a Cargo lock deadlock.

The first large-board generation also exposed two file-backend limits: an unbounded read `JoinSet` could exceed common per-process file-descriptor limits, and item-by-item import revalidated the growing board on every save. The implementation now bounds concurrent reads at 64 and uses one validated batch write plus one issued-ID history update for file-backend imports. Each benchmark sample uses a fresh temporary board, so reset/import work is outside the measured command timings.
