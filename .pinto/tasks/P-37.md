# **Summary**

`src/service/doctor.rs` (1,479 lines), `src/service/sprint.rs` (1,430 lines), and `src/storage/file_repository.rs` (1,320 lines) remain the largest source files. Each still has an internal split axis: inspection versus repair in doctor, lifecycle versus capacity and reports in the sprint service, and item versus sprint persistence in the file repository.

# **Acceptance Criteria**

- [x] The three files are split into focused submodules following the Rust 2018 layout used by earlier splits.
- [x] Public paths and behavior are unchanged and the full test suite passes.
- [x] `mise run check` passes.

# **Implementation Plan**

Follow the Rust 2018 `parent.rs` + `parent/` layout already used by `service/item` and `storage/sqlite_repository`: the parent file keeps shared types and the public surface, submodules hold the concern-specific bodies, and tests move into a `tests.rs` submodule. Public paths and behavior stay unchanged; correctness is guaranteed by the existing test suite staying green after each split.

## `service/doctor.rs` → `service/doctor/` (inspection vs repair)

The raw-record data model (`Inspection`, `RecordArea`, `RawField`, `RawItemRecord`,
`RawSprintRecord`, `IssuedHistory`, `string_field`, `string_list_field`) stays in the parent
module rather than a separate `record` submodule, because both `inspect` and `repair` read
these structs' private fields. Keeping them in the parent lets the submodules touch those
fields without widening any visibility (Rust makes a module's private items visible to its
descendants). Inspection functions that the test module exercises directly are re-imported
into the parent so `use super::*` continues to resolve them.

- `doctor.rs` (parent): shared report types (`DoctorIssueKind`, `DoctorIssue`, `DoctorFix`, `DoctorReport`), entry points (`doctor`, `run_doctor`, `run_doctor_with`), the raw-record data model listed above, submodule declarations, and `mod tests`.
- `doctor/inspect.rs`: board reading and analysis (`inspect_board`, `inspect_file_storage`, `inspect_sqlite_storage`, `read_documents`, `read_issued_history`, `make_issue`, `analyze_sprints`, `analyze_records`, `analyze_issued`, `graph_cycles`, `visit_graph`).
- `doctor/repair.rs`: safe fixes (`apply_safe_fixes`, `DuplicateRepair`, `DuplicateRepairPlan`, `repair_duplicate_item_ids`, `plan_duplicate_item_repairs`, `record_area_priority`).
- `doctor/tests.rs`: the existing `#[cfg(test)] mod tests`.

## `service/sprint.rs` → `service/sprint/` (lifecycle vs capacity/reports)

- `sprint.rs` (parent): shared types (`SprintLoadWarningKind`, `SprintLoadWarning`, `SprintCloseAction`), submodule declarations, re-exports, and `mod tests`.
- `sprint/lifecycle.rs`: create/edit/delete/start/close/transition and assignment (`create_sprint`, `edit_sprint`, `delete_sprint`, `start_sprint`, `close_sprint`, `transition_sprint`, `rollback_sprint_close`, `validate_sprint_assignment`, `assign_sprint`, `assign_sprint_raw`, `assign_sprint_by_status`, `rollback_bulk_assignment`, `unassign_sprint`).
- `sprint/capacity.rs`: capacity and reports (`list_sprints`, `set_sprint_capacity`, `sprint_capacity`, `sprint_load_warnings`, `sprint_load_warnings_for`, `historical_velocity_threshold`).
- `sprint/tests.rs`: the existing `#[cfg(test)] mod tests`.

## `storage/file_repository.rs` → `storage/file_repository/` (item vs sprint persistence)

Mirror the `sqlite_repository` split.

- `file_repository.rs` (parent): `FileRepository` struct, constructor and path helpers, shared helpers (`markdown_paths`, `is_markdown`), submodule declarations, and `mod tests`.
- `file_repository/items.rs`: `impl BacklogItemRepository for FileRepository` plus item helpers (`read_all_item_records`, `read_item_records`, `ensure_unique_item_ids`, `validate_item_filename`).
- `file_repository/sprints.rs`: `impl SprintRepository for FileRepository` plus sprint helpers (`read_sprint_records`, `ensure_unique_sprint_ids`, `validate_sprint_filename`).
- `file_repository/tests.rs`: the existing `#[cfg(test)] mod tests`.
