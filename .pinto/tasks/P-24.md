# **Summary**

Align release metadata with the actual publication state and make SQLite schema incompatibility visible before release.

# **Implementation Strategy**

## Release state contract

- Keep the package version, every committed lockfile entry for pinto-cli, the latest release tag, and published installation examples consistent.
- Represent work that has not been published as an explicit `[Unreleased]` CHANGELOG section instead of a dated release heading.
- Add a portable release metadata checker that fails on version, lockfile, installation-example, CHANGELOG, or tag mismatches.

## SQLite compatibility documentation

- Document the v1-to-v2 schema change in the stability guidance, including affected users, observed symptoms, backup requirements, and downgrade or recovery steps.
- Keep the existing fail-fast behavior and make the version-2 compatibility boundary and no-automatic-migration rule explicit.

## Release gate and verification

- Run the metadata and compatibility-document checks from `mise run release-check` and the CI release job before packaging or publishing.
- Make the canonical `release-publish` task depend on the complete release gate so publishing cannot proceed through the project release path before those checks pass.
- Add integration coverage for the passing contract and each mismatch class.
- Follow Red → Green → Refactor, run the complete relevant checks, and perform iterative review before updating this item.

# **Acceptance Criteria**

- [x] Unreleased version metadata remains clearly unreleased until the public release is made, and published version references stay consistent.
- [x] The SQLite schema v1-to-v2 breaking change documents affected users, symptoms, backup requirements, and downgrade or recovery guidance.
- [x] A release check detects mismatches among package versions, lockfiles, installation examples, changelog headings, and tags.
- [x] Public release is blocked until the breaking-change documentation and version checks are complete.
