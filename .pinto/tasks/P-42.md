# **Summary**

The board format is designed for Git, yet no documentation covers merging boards edited in parallel branches or clones. Sequential IDs collide, `issued_ids` produces textual conflicts, and merged boards can contain duplicate IDs without a documented recovery path.

# Implementation Plan

## Approach

Add a new mdBook page `docs/book/src/merging.md` — a runbook for merging boards edited in parallel clones/branches. Link it from `SUMMARY.md` under "Using pinto" and register it in the maintained-guidance list of `tests/docs.rs`.

## Documented mechanics (verified by dogfooding)

- pinto allocates the next sequential ID from `issued_ids`. Two clones branching off the same base each allocate the same next ID (e.g. both `T-2`), so a merge produces an add/add conflict in `.pinto/tasks/T-2.md`. When the branches diverge by different counts, `.pinto/issued_ids` also takes a textual (content) conflict on the appended lines.
- Resolve `issued_ids` by taking the **union** of both sides (append-only history — never drop an issued number).
- Resolve the task-file conflict by keeping one item for the shared ID and re-homing the other under the next free ID via `pinto add`, which allocates beyond the merged `issued_ids`.
- Verify with `pinto doctor`: it reports duplicate IDs, filename/ID mismatches, and `issued_ids` history gaps with explicit repair directions. `pinto doctor --fix` backfills a missing `issued_ids` entry but never chooses between duplicates. Re-run until `Board is healthy.`
