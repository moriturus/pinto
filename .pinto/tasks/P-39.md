# **Summary**

`rm --force` is irreversible and file-backend boards keep no history, so recovering from a mistaken mutation falls back to manual Git commands at best. Provide a guided undo for the most recent completed board mutation, at minimum on the Git backend.

# **Acceptance Criteria**

- [x] A feature decision record covers scope, per-backend behavior, and compatibility impact.
- [x] On the Git backend, one documented command reverts the most recent completed board mutation.
- [x] On backends without history, the command fails with an actionable message describing the available recovery options.

# **Implementation Plan**

## Command surface

Add a top-level `undo` subcommand that reverts the most recent completed board
mutation.

- `pinto undo` takes no positional arguments; it targets the single most recent
  completed board mutation.
- Scope is deliberately narrow: one level of undo (the last mutation), not an
  unbounded history walk. This keeps the command lightweight and predictable;
  deeper recovery stays the user's Git responsibility.
- `undo` is excluded from `pinto automate` plans (added to the unsafe command
  set): reverting a mutation is a human corrective action, not an agent-plan
  step.

## Per-backend behavior

pinto only records history on the Git backend, so undo is backend-specific:

- **Git backend**: each board mutation is one `pinto: <verb> <id>` commit. `undo`
  reverts the current `HEAD` with `git revert --no-edit HEAD`, producing a new
  commit that reverses the change. Revert (not reset) is chosen because it is
  non-destructive and Git-friendly: history is preserved and the undo itself is
  reviewable with `git diff` and reversible.
  - `undo` refuses when `HEAD` is not a pinto board mutation (subject does not
    start with `pinto: `) or the repository has no commits, with a message
    pointing at `git log -- .pinto` and manual `git revert <sha>`. This prevents
    undo from silently reversing an unrelated user commit stacked on top of the
    board.
- **File / SQLite backends**: no history is kept. `undo` fails fast with an
  actionable message that names the current backend, explains it keeps no
  history, and lists recovery options (restore from a backup or VCS, or switch to
  `[storage] backend = "git"` to enable undo for future mutations).

## Layering

- `GitRepository::undo_last()` runs the revert through the existing `git`
  subprocess helpers, reusing `fallback_identity_args` so a revert commit
  succeeds even without a configured Git identity. It returns the reverted commit
  subject for the CLI message. Refusals surface as `Error::UndoUnavailable`.
- `Backend::undo()` dispatches: Git delegates to `undo_last`; File/SQLite return
  `Error::UndoUnsupported`, which names the backend and the recovery options.
- Service `undo_last_mutation(project_dir)` opens the board under the write lock
  (same serialization as other mutations) and returns an `UndoOutcome`.
- CLI `cmd_undo` renders the reverted subject on success and maps the
  no-history/refusal errors to exit code 1.

## Compatibility / persistence

Additive: a new read-then-write command, no data-format or schema change and no
migration. It reuses the plain-text persistence and the Git commit convention. A
feature decision record covering scope, per-backend behavior, and compatibility
lives in `docs/book/src/undo.md` and this plan.
