# **Summary**

Expose the validated automation-plan JSON Schema through pinto automate --schema so agents and CI can inspect the accepted plan shape before running it.

# **Implementation Plan**

1. Keep `AutomationPlan::parse` as the runtime source of truth and expose one JSON Schema definition from the automation domain module.
2. Constrain the schema to the object and command-array shape accepted by the parser, reject unknown properties, and enumerate only non-recursive, non-interactive top-level command names and their safe CLI aliases. Existing clap parsing remains authoritative for command arguments.
3. Add `automate --schema` as a plan-independent, mutually exclusive CLI mode that prints only machine-readable JSON Schema and does not require a board.
4. Add unit and CLI integration tests for schema shape, unknown fields, unsafe commands and aliases, plan independence, localization, and regression coverage for existing plan modes.
5. Document the schema contract in the CLI/user documentation and update the automation-plan demo with a reproducible schema inspection command.

# **Acceptance Criteria**

- [x] `pinto automate --schema` prints a machine-readable JSON Schema without requiring an execution plan.
- [x] The schema describes the accepted command-array structure and rejects unknown fields or unsafe command forms consistently with the validator.
- [x] The schema documentation and validation tests remain synchronized with the supported automation plan.
- [x] Existing `--plan`, `--dry-run`, and `--json` automation behavior remains unchanged.
