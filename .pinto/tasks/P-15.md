# **Summary**

Complete the archive workflow by listing and viewing archived PBIs and restoring them safely to the active backlog.

# **Implementation Approach**

## Repository boundary

- Extend the backlog-item repository with explicit archived-list, archived-load, and restore operations. Implement the same contract for file, Git, and SQLite backends so the service layer stays backend-independent.
- Preserve the existing active-only list/load behavior and keep archived records in their original content, ID, rank, and relationships.

## CLI and service behavior

- Add `--archived` to `list` and `show`; the option selects archived PBIs, applies the existing list filters and ordering, and leaves normal active views unchanged.
- Add `restore <id>` as a write operation that moves the archived record back to the active store and commits the Git-backed mutation when applicable.

## Collision safety

- Check the active destination before restoring and return an actionable error when the ID is already active. Never overwrite either copy. File restoration moves the archive document only after validation; SQLite clears its archived flag without changing the row.

## Verification

- Follow Red → Green → Refactor with repository, service, and CLI integration tests for archive listing, archived details, restoration fidelity, active-view exclusion, and ID-collision refusal. Add an archive-recovery demo and document the commands.

# **Acceptance Criteria**

- [x] `pinto list --archived` lists archived PBIs, and `pinto show <id> --archived` displays an archived item's details.
- [x] `pinto restore <id>` returns an archived PBI to the active task store without changing its content or ID.
- [x] Normal list, board, and show operations continue to exclude archived PBIs unless the archive option is supplied.
- [x] Restoration refuses an ID collision without overwriting either copy.
