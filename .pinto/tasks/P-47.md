# **Summary**

Make configurable Kanban modified-key bindings work reliably across terminal event encodings, including the documented `Ctrl+?` regex-search shortcut, other `Ctrl+` combinations, and bindings using other supported modifiers. The fix and verification should cover the shared key-matching path rather than treating one shortcut or modifier as an isolated case.

# **Implementation Plan**

- Keep terminal representation normalization in the shared, platform-neutral `KeyStroke` matcher invoked by the crossterm Kanban keymap, without OS-specific runtime branches.
- Preserve every non-Control modifier bit while matching Unix control-byte aliases: control-letter case differences, `Ctrl+@`/`Ctrl+[`/`Ctrl+\`/`Ctrl+]`/`Ctrl+^`/`Ctrl+_`, and DEL/`Backspace` for `Ctrl+?`.
- Add Red竊竪reen竊坦efactor coverage for `Ctrl+?`, other `Ctrl+` combinations, combined modifiers, named `Shift` keys, and negative cases so `?` and `Ctrl+?` do not absorb unrelated plain or control bindings.
- Run focused tests, the repository verification gate, and PTY dogfooding on the available platform; record the platform-dependent evidence and limits below.

# **Acceptance Criteria**

- [x] `Ctrl+?` opens regex search from the documented default shortcut in a standard terminal.
- [x] Other configured `Ctrl+` key bindings are recognized through the same Kanban input path in a standard terminal.
- [x] Configured bindings using other supported modifiers are recognized through the same Kanban input path in a standard terminal.
- [x] Automated regression coverage exercises `Ctrl+?`, other `Ctrl+` combinations, and other supported modifiers, including the terminal event representations under test.
- [x] `?` still opens help and existing plain-key bindings keep their current behavior.
