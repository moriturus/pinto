# **Summary**

Provide a consistent JSON board export for automation without making ordinary reads block on board-wide locking.

## Implementation Plan

1. Acquire the board write lock before opening `config.toml` or the selected storage backend in `export_snapshot`.
2. Keep the lock guard alive while reading active PBIs, Sprints, effective configuration, and the shared Definition of Done, then return the assembled snapshot.
3. Leave ordinary read commands non-blocking, document that they do not provide board-wide snapshot isolation, and direct shell or agent automation that needs consistency to `export --json`.
4. Add concurrency coverage for exports alongside Sprint close and rank rebalance, update the export demo and user documentation, then verify the behavior through the CLI.

## Consistency Contract

`export --json` is the consistent active-board read path: it waits for a writer and prevents a Sprint close or rebalance from changing records during the export. `list`, `show`, `board`, and other ordinary reads remain available without the board-wide lock and may observe different records when a writer runs concurrently.

## Verification Plan

- Add failing integration tests for export serialization against Sprint close and rebalance before implementing the lock.
- Run focused tests, `mise run check`, and the export demo.
- Inspect the export and backlog through `cargo run`, complete every Acceptance Criteria checkbox, and commit with the P-25 ID.

# **Acceptance Criteria**

- [x] export --json acquires the board lock before opening configuration and storage and holds it until the export completes.
- [x] Concurrency tests prove that an export cannot combine old and new records while sprint close or rebalance is writing.
- [x] Ordinary read commands remain non-blocking and their lack of board-wide snapshot isolation is documented.
- [x] Shell and agent automation that requires a consistent board is directed to export --json.
