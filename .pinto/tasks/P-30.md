# **Summary**

Keep localized English errors and library display messages consistent without altering diagnostics supplied by external systems.

## Implementation Plan

- Treat `Error::to_string()` as the stable English library fallback and `Error::localized()` as the CLI/TUI localization boundary.
- Make the English Fluent entry for every public `Error` variant semantically and textually equivalent to its `Display` message, while interpolating OS, Git, TOML, parser, and other external diagnostics unchanged.
- Preserve structured `Error` values through service and command layers; localize only when rendering human-facing output, and keep JSON error fields structured/localized without rewriting source diagnostics.
- Add unit and CLI integration coverage for all variant mappings, representative recovery guidance, and external diagnostic preservation, plus an i18n demo update.

# **Acceptance Criteria**

- [x] The intended relationship between localized English messages and `Error::to_string()` is defined for every public error.
- [x] Representative errors are tested for matching meaning and actionable recovery guidance.
- [x] Structured error information remains available until the localization boundary where possible.
- [x] OS, Git, TOML, and other external diagnostics retain their original text.
