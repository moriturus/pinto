# **Summary**

Add `pinto doctor` to diagnose board integrity problems and offer only safe, explicit repairs for recoverable cases.

## Implementation Plan

1. Add a storage-aware integrity snapshot that reads active/archive Markdown records without stopping at the first malformed record, then validates references, relation cycles, IDs, workflow statuses, rank normal form/uniqueness, and tasks/archive collisions with path-level diagnostics.
2. Keep automatic repair conservative: append valid existing IDs missing from `issued_ids` and rename a valid record file to its frontmatter ID only when the destination is unoccupied; leave relationship, duplicate, status, rank, and collision issues for explicit manual repair so no board data is discarded.
3. Expose `pinto doctor [--fix]` through the service and CLI. Re-scan after fixes, print every applied change and every unresolved issue with an actionable repair direction, and return success only when the final board is clean.
4. Add domain/service and CLI integration coverage plus a small doctor demo, and verify the command through `cargo run` on clean and intentionally damaged temporary boards.
5. Repair the release package baseline for `src/service/next.rs` and make `verify-package.sh` check every current `src/**` file directly, so adding a runtime source file no longer creates a stale-baseline release failure; document the updated release check.

# **Acceptance Criteria**

- [x] `pinto doctor` detects dangling dependency, parent, and Sprint references; relation cycles; duplicate IDs; issued-ID inconsistencies; invalid statuses; rank anomalies; and tasks/archive collisions.
- [x] Each reported issue identifies the relevant path or PBI and gives an actionable repair direction.
- [x] The command returns a non-zero exit status when integrity problems are found and a zero status for a clean board.
- [x] `pinto doctor --fix` applies only safe mechanical repairs, reports every change, and never silently discards board data.
